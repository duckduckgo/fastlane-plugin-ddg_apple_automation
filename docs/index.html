What start_new_release does

  3. Updates embedded files (e.g. privacy config) on the branch
  4. Updates the version xcconfig file with the new version
  5. Calls find_release_dri to determine the Asana user to assign the task to
  6. Creates a new Asana release task from template, assigned to the release DRI
  7. Updates the Asana release task with the commit log since the last public release (tasks included in this release)

⚠️ Branch Creation Failures

What can go wrong: The release branch already exists (from a previous failed run), or there is a version conflict.

Recovery: Check whether the platform release branch (e.g. release/ios/<version> or release/macos/<version>) exists in GitHub. If it is stale, delete it. Then re-run the workflow.

⚠️ Signing & Build Failures

What can go wrong: Code signing sync failure (the workflow has retry logic but can still fail), archive or notarization timeout.

Auto: asana_log_message logs build status to the Asana release task regardless of outcome.

Recovery: Check match repo SSH access. Verify that the MATCH_PASSWORD and SSH_PRIVATE_KEY_FASTLANE_MATCH secrets are current. Re-run the failed jobs in GitHub Actions.

⚠️ Tag & Merge Failures

What can go wrong: Merge conflicts between the release branch and main, or the tag already exists from a previous run.

Auto: asana_create_action_item with template internal-release-tag-failed — creates a subtask assigned to the release DRI with manual merge instructions. (Only fires on internal-release-bump context; not on code freeze.)

Recovery (macOS): Resolve conflicts locally on the release branch and push. Then manually trigger macos_tag_release.yml with the release branch name and the Asana task URL as inputs.

Recovery (iOS): Resolve conflicts locally on the release branch and push. Then manually trigger ios_tag_release_update_asana.yml with the release branch name and the Asana task URL as inputs.

⚠️ Publish & Appcast Failures macOS

What can go wrong: Appcast generation failure (malformed XML, missing DMG), S3 upload failure, or Asana update failure.

Auto: asana_create_action_item with template appcast-failed-internal — notifies the DRI that the appcast step failed and includes recovery instructions.

Recovery: Verify S3 credentials (AWS_ACCESS_KEY_ID_RELEASE_S3, AWS_SECRET_ACCESS_KEY_RELEASE_S3). Check appcast XML format. Manually trigger macos_publish_dmg_release.yml with inputs: asana-task-url, tag (from the tag_and_merge step), release-type: internal.

⚠️ Metadata Upload Failures iOS

What can go wrong: App Store Connect API authentication failure, network timeout, or S3 upload error during upload_current_metadata.

Recovery: Verify that APPLE_API_KEY_ID, APPLE_API_KEY_ISSUER, and APPLE_API_KEY_BASE64 secrets are current. Verify AWS credentials (AWS_ACCESS_KEY_ID_RELEASE_S3, AWS_SECRET_ACCESS_KEY_RELEASE_S3). The metadata upload runs in parallel with create_release_branch, so a failure here does not block branch creation — manually trigger ios_upload_version_metadata.yml to retry.

▶ validate_input_conditions validate_internal_release_bump
Runnermacos-26
Timeout10 minutes

Steps

  1. Asserts the workflow is running on a release branch (or main)
  2. Finds or verifies the release task in Asana — searches the Apple Releases section for a task matching the current marketing version; only considers tasks created in the last 5 days
  3. Checks the branch is NOT frozen — errors if a draft GitHub release named <version>+<platform> already exists (draft public release is the freeze indicator)
  4. Verifies code changes exist since the last tag — ignores changes under .github/, scripts/, fastlane/ directories; sets skip_release=true if no relevant changes are found
  5. Validates release notes are not empty and don't contain only placeholder text (<-- Add release notes here -->)
  6. Reports if an active hotfix task exists in the Apple Releases section — blocks normal bump and adds the release task assignee as a collaborator on the hotfix task

Both platforms call validate_internal_release_bump with their respective platform: argument; the validation logic is identical.

⚠️ Validation Failures

Frozen branch: validate_internal_release_bump errors if a draft GitHub release named <version>+ios (or +macos) already exists. This means a public release has been decided. No further internal bumps are allowed until the public release is tagged (which removes the draft and unfreezes the branch).

Empty release notes: The bump is blocked. Someone must fill in release notes in the Asana release task before the internal bump can proceed.

No code changes: skip-release is set to true and the workflow exits gracefully — this is NOT an error; it's expected when only non-code files changed (e.g. workflow configs, scripts, fastlane files).

Active hotfix: If an active hotfix task is found in the Asana Apple Releases section, the normal internal bump is blocked to prevent conflicts with the in-progress hotfix. The release task assignee is automatically added as a collaborator on the hotfix task.

App Store Submission

Promotes the latest TestFlight build to the App Store and freezes the release branch to prevent further internal bumps. This is a manual step performed by the release DRI on Friday (iOS) or Thursday (macOS), after the internal build has been tested during the week.

macOS is submitted a day earlier because App Store review for macOS takes considerably longer than iOS — often several days, sometimes not completing until Tuesday. Submitting on Thursday gives the build more time in the review queue, increasing the chances of approval by Monday. If the release needs another bump after submission (e.g. on Friday), the build is removed from review, the branch is unfrozen, the release is bumped and resubmitted.

Full Rollout

After the public release on Monday, the App Store phased rollout starts at 5%. On Wednesday, the release DRI accelerates the rollout to 100% of users via App Store Connect — provided the build is stable. On macOS, the Upload Version Metadata workflow is run a few hours after accelerating the rollout.

Manual — App Store Connect — no GitHub Actions workflow required

▶ Accelerate Rollout to 100% Both platforms manual — App Store Connect

Monitor the staged rollout at 5%. Check for crashes and emergency feedback.

Once the build is confirmed stable and it is not late in the week (before Thursday), release to 100% of users via App Store Connect.

Steps

  1. Open App Store Connect for the relevant platform
  2. Navigate to the current version
  3. Click Release to All Users (or equivalent) to accelerate the phased rollout to 100%

If delayed past mid-week

  • Option A: Pause the rollout and wait until the following week to release to 100% — minimises weekend exposure.
  • Option B: Release to 100% but remain on deck to monitor incidents. Pay particular attention to Grafana crash dashboards and other immediate feedback channels before the weekend.

Release Branch Freeze Mechanism Both platforms

The freeze mechanism prevents further internal releases once a version is ready for public release. Without it, any push to the release branch would trigger another internal bump, potentially overwriting the build that QA has already validated.

The freeze mechanism works identically for both platforms. The freeze_release_branch action creates a draft GitHub release tagged with the platform suffix (e.g. 7.5.0+ios or 7.5.0+macos). Both ios_promote_testflight.yml and macos_promote_testflight.yml can trigger freezing via their freeze-release input.
  1. Freezing. Triggered automatically by ios_promote_testflight.yml or macos_promote_testflight.yml when freeze-release is true (the default). After promoting a TestFlight build to the App Store, the action freeze_release_branch runs and creates a draft GitHub release named <version>+<platform> (e.g., 7.5.0+ios or 7.5.0+macos). This draft release is never published — it exists purely as a machine-readable signal.
  2. Blocking. When validate_internal_release_bump runs on a push to the release branch, it calls GitHelper.assert_release_branch_is_not_frozen, which checks for the presence of that draft release. If found, it sets the skip_release GHA output to true and the workflow exits without building a new internal release.
  3. Unfreezing (automatic). When tag_release runs for a public release (is_prerelease: false), it calls GitHelper.unfreeze_release_branch before creating the real public tag. This deletes the draft release, unfreezing the branch as part of the normal public release flow.
  4. Unfreezing (manual). If you need to allow internal bumps again — for example, because the public release was delayed — go to GitHub → Releases and delete the draft release named <version>+<platform> manually. After the additional changes have been pushed and the internal bump completes, re-freeze the branch by running the promote TestFlight workflow for the relevant platform (ios_promote_testflight.yml or macos_promote_testflight.yml) — see App Store Submission.
Flow summary:
Promote to App Store → Draft release created (frozen) → Push to release branch skipped → Public release tags → draft deleted (unfrozen)

Sparkle / Appcast Pipeline macOS only

macOS uses the Sparkle framework for over-the-air updates. The pipeline generates and publishes an appcast XML file to S3 so that existing installs can discover new builds.

iOS has no equivalent. iOS distributes updates exclusively through TestFlight and the App Store. There is no Sparkle appcast, no DMG, and no S3 update feed for the iOS app.

Sparkle Tools

The sign_update and generate_appcast command-line tools are downloaded at build time from the sparkle-project/Sparkle GitHub releases. They are used for signing DMG entries with the EdDSA private key and regenerating the appcast XML.

appcastManager.swift

A Swift script at macOS/scripts/appcast_manager/appcastManager.swift that manages appcast XML generation. It has three operating modes:

  • --release-to-internal-channel — used by the code freeze and internal bump flows. Adds the new DMG to the internal Sparkle channel (appcast URL for the internal config).
  • --release-to-public-channel — used for public releases. Promotes the already-published internal entry to the public appcast URL.
  • --release-hotfix-to-public-channel — used for hotfix releases. Publishes a new DMG entry directly to the public channel, bypassing the internal channel.

Publishing Flow

  1. Fetch the current appcast XML from S3.
  2. Back up the existing appcast (uploaded as a workflow artifact for rollback).
  3. Download the DMG from S3.
  4. Extract release notes from the Asana release task body using the fetch_release_notes helper. Notes must not be empty or placeholder — the publish step validates this before proceeding.
  5. Run appcastManager.swift with the appropriate mode to sign the DMG entry and generate the updated appcast XML.
  6. Upload the new appcast to S3, replacing the previous version.

Post-publish Validation

After a successful publish, an Asana action-item subtask is created using the validate-check-for-updates-internal or validate-check-for-updates-public template. The appcast diff and release notes are attached to the task for manual verification before the release is considered complete.

Appcast URLs

Appcast feed URLs are stored in macOS/Configuration/App/Sparkle.xcconfig under the pattern SPARKLE_URL_<CONFIG>. Release automation reads these values to determine the correct S3 target for each build configuration:

  • SPARKLE_URL_RELEASE — public appcast: staticcdn.duckduckgo.com/macos-desktop-browser/appcast2.xml
  • SPARKLE_URL_ALPHA — alpha appcast: staticcdn.duckduckgo.com/macos-desktop-browser/alpha/appcast-alpha.xml
